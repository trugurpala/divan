# Divan v0.15.0 Portable Project OS Specification

## Purpose

Divan Project OS carries a user-authorized intent from specification through implementation evidence, preview, release, live observation, and applicable SEO checks. It is a portable, local-first contract for an installed project; it is not a hosted control plane, autonomous company, or background service.

The user remains the **Padişah**: they issue intent, approve material scope changes, and retain authority over credentials, publication, and destructive actions. **Divan** is the functional council: deterministic roles route work, produce plans and receipts, and report what is blocked rather than simulating success. Supervised autonomy permits only a bounded, authorized, evidence-producing step; it never widens authority, installs software, deploys, or persists secrets by itself.

## Functional governance

| Function | Responsibility | Boundary |
|---|---|---|
| Padişah (user) | States intent; approves scope and external effects | Final authority; may pause or reject any phase |
| Sadrazam/controller | Normalizes intent and composes a deterministic plan | No credentials, hidden inference, or background loop |
| Divan roles | Product, engineering, quality, delivery, documentation | Only approved plans and declared provider capabilities |
| Providers | GitHub, Vercel, Context7, local tools | Declare operations and evidence; absence is `BLOCKED` |
| Receipts | Append-only, redacted phase evidence | Never contain secrets, personal absolute paths, or hidden reasoning |

Role names are reader-facing. Machine-facing IDs, JSON keys, filenames, and CLI flags are English and UTF-8; Turkish is a first-class locale rather than a duplicate implementation.

## Two-layer standards

1. **DCS-*** (Divan Community Standards) govern the Divan distribution: licensing, release, repository hygiene, host compatibility, and public surfaces.
2. **DPS-*** (Divan Project Standards) govern an initialized target project: applicability, project evidence, delivery checks, waivers, and receipts.

A DCS rule is not silently imposed on a target project and a DPS rule does not weaken Divan's release gates. Each DPS rule declares applicability for `library`, `service`, `application`, `public-web`, `documentation`, or `monorepo`; an inapplicable rule is reported, not failed.

## State machine and evidence

Goals use the canonical states `DISCOVERED`, `SPECIFIED`, `PLANNED`,
`IMPLEMENTING`, `VERIFIED`, `PREVIEWED`, `RELEASED`, `OBSERVED`, `BLOCKED`,
and `FAILED`. The complete transition contract is:

| From | Permitted next states |
|---|---|
| `DISCOVERED` | `SPECIFIED`, `BLOCKED`, `FAILED` |
| `SPECIFIED` | `PLANNED`, `BLOCKED`, `FAILED` |
| `PLANNED` | `IMPLEMENTING`, `BLOCKED`, `FAILED` |
| `IMPLEMENTING` | `VERIFIED`, `BLOCKED`, `FAILED` |
| `VERIFIED` | `PREVIEWED`, `RELEASED`, `BLOCKED`, `FAILED` |
| `PREVIEWED` | `RELEASED`, `BLOCKED`, `FAILED` |
| `RELEASED` | `OBSERVED`, `BLOCKED`, `FAILED` |

`BLOCKED` names a missing approval, provider capability, prerequisite, or
evidence and records `resume_from` as the immediately preceding nonterminal
phase. After the missing condition is resolved, `goal resume` may return only
to that `resume_from` phase and emits a new receipt. `FAILED` and `OBSERVED`
are terminal; retry after `FAILED` requires a new goal. Legal transitions are
verified by append-only, redacted, project-relative receipts. Release is
complete only after provider evidence and live readback support `OBSERVED`.

## Public interfaces

| Interface | Contract |
|---|---|
| `divan init` | Produces a dry-run write plan by default; `--execute` is required to write managed project files. |
| `divan audit` | Inspects bounded project evidence without executing detected project code. |
| `divan verify` | Reports applicable project standards as `PASS`, `FAIL`, `BLOCKED`, or `NOT_APPLICABLE`. |
| `divan goal start|status|resume` | Creates or reads deterministic goal artifacts and preserves state-machine legality. |
| `divan receipt verify` | Verifies schema, hashes, redaction, transitions, and project-relative paths. |
| `divan release` | Is dry-run by default and requires `--execute`, provider capability, release evidence, and live readback. |

Canonical JSON is stable UTF-8 with a trailing newline. Existing Turkish entry points remain narrow deprecated wrappers in v0.x and contain no duplicate business logic.

## Provider boundary

Providers are capability declarations, not ambient authority. A capability contains an ID, availability, allowed operations, missing requirements, and safe evidence. `local` owns local inspection and deterministic plans; `github` owns PR, CI, tag, release, and remote readback evidence; `context7` supplies current official framework documentation evidence; and `vercel` owns preview, browser verification, staged production, and promotion evidence.

Credentials exist only in the selected provider environment and are never written into plans or receipts. If an operation has no provider, permission, or evidence route, Project OS returns `BLOCKED`; it does not manufacture a green result.

## Safety and operating constraints

- All mutations are dry-run-first and require `--execute`.
- Inspection does not execute target-project code and bounds traversal, file size, symlink behavior, workspace count, and depth.
- Managed-file updates are atomic, project-contained, and preserve content outside explicit markers. Unmatched or duplicate markers fail closed.
- Every changed path is classified or explicitly reported as `unclassified`.
- No daemon, scheduler, hosted dashboard, telemetry service, or continuously running agent is part of Project OS. Each invocation exits after emitting its plan, result, or receipt.
- Candidate `ADOPT`/`ADAPT` status does not install, vendor, or run code. A later adoption requires its own immutable pin, attribution, license review, tests/evals, and inspection.

## Non-goals

v0.15.0 does not promise autonomous product selection, unattended releases, cross-project memory, automatic credential discovery, remote state storage, or measured quality improvement. Those claims need separate authority and, where applicable, real-agent evaluation under `evals/README.md`.
